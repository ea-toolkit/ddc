# Cycle 008: Work Orders Not Released for Picking

## Problem Input

Work orders are not being released for picking at fulfillment systems. Warehouses have inventory but no pick instructions are arriving.

## Agent Before (Zero Context)

The KB had good coverage of the "orders not dropping" pattern and the fulfillment chain (Service Order Manager → OrderIntegrationHub → WarehouseManagementSystem). However, the agent had critical gaps:

- Did not know what a "work order" was or how it differed from a sales order
- Did not know the External Routing Provider generates picking instructions (treated it as a routing pass-through)
- Did not know the Service Order Manager → External Routing Provider → warehouse path for pick instruction generation
- Did not know whether release was manual or automatic
- Did not know whether wave planning or batching existed at this level

**Confidence: 2/5** — Could list possible failure points from the existing "orders not dropping" pattern but couldn't pinpoint the failure layer or reason about the work order release flow specifically.

**Key mistake in first attempt:** Asked the user "what was the actual root cause?" and "what do the systems show?" — diagnostic questions the agent should answer, not ask. The user correctly pushed back: the demand checklist should ask for domain knowledge gaps, not for the answer itself.

## Human Answers

The user provided comprehensive domain knowledge covering:

- **Terminology**: Work order = SaveWorkOrder message from Service Order Manager to External Routing Provider. Release = pick tasks visible on screens. Automatic, no manual step.
- **Flow**: Order placed → Service Order Manager → SaveWorkOrder via MessageBroker → External Routing Provider stores in DB → generates picking instructions → warehouse systems → fulfillment screens.
- **No batching**: Real-time, individual order flow. No wave planning at this level.
- **Preconditions**: Met by the time Service Order Manager sends the message.
- **Failure points**: Four-layer diagnostic from Service Order Manager outbound through External Routing Provider internals to warehouse delivery.
- **Key insight**: The problem statement ("warehouses have inventory but no pick instructions") points to the External Routing Provider layer — orders exist upstream, the break is in the instruction generation chain.

The user also corrected the agent's approach: the demand checklist should not ask for the root cause or diagnostic observations — those are the agent's job. The checklist should only request domain knowledge needed to reason about the problem.

## Agent After (With Context)

**Confidence: 4/5**

With the curated knowledge, the agent diagnosed the failure layer:

1. **Ruled out inventory issues** — warehouses have inventory, so no backordered/ATP corruption pattern.
2. **Identified the External Routing Provider as the failure layer** — it generates picking instructions, making it the critical link. The Service Order Manager likely sent SaveWorkOrder messages successfully (orders exist upstream). The break is at or after the External Routing Provider.
3. **Most likely root cause**: External Routing Provider receiving work orders into its database but failing internal processing to generate picking instructions (routing logic error, database issue, or platform issue).
4. **Alternative causes**: MessageBroker queue consumer on External Routing Provider side is down (messages piling up), or External Routing Provider generating instructions but downstream integration to warehouse systems is broken.
5. **Classified as a new variant** of the "orders not dropping" pattern — root cause #9: messages delivered but recipient failed to act on them.

**What improved**: Understanding that the External Routing Provider is not just a routing pass-through but the system that generates picking instructions. This was the key insight that allowed pinpointing the failure layer.
